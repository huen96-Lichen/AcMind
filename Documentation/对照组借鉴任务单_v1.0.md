# 对照组借鉴任务单

> **版本**: v1.0  
> **日期**: 2026-06-13  
> **目标**: 在尽量不原创、优先借鉴对照组成熟模式的前提下，提升 AcMind 的完成度、统一性与可感知 UI 质量。

---

## 1. 执行前提

本任务单默认以下策略：

- 优先复用 AcMind 现有代码，不重做已有能力
- 优先吸收对照组成熟模式，不做新概念发明
- UI 以统一壳层、首屏可读、真实状态为准
- 所有任务都必须能回指到明确的代码文件或模块
- 任何“看起来更酷”的方案，如果会增加结构分裂，一律降级处理

## 3. 任务清单

### T01. 配置迁移链标准化

- **优先级**: P0
- **借鉴来源**: HermesPet - `SchemaMigrator`
- **AcMind 对标模块**: `AcMindKit/Services/Storage/DataMigrationService.swift`、`AcMindKit/Services/Settings/SettingsService.swift`
- **目标**: 把散点式配置演进收束成可审计、可回滚、幂等的迁移链
- **任务内容**:
  - 引入统一版本号
  - 为 settings / voice / hotkey / local preferences 增加迁移入口
  - 把 key 变更、语义变更、默认值变更拆成独立迁移
  - 写一次性迁移完成标记
- **交付物**:
  - 迁移框架
  - 至少 3 条真实迁移
  - 回归测试
- **预估工期**: 1 - 2 天
- **依赖**: 无
- **验收标准**:
  - 老数据升级不丢配置
  - 二次执行不会重复破坏数据
  - 新增迁移有明确版本记录

### T02. 结构化 Logger 统一

- **优先级**: P1
- **借鉴来源**: boring.notch - `Logger`
- **AcMind 对标模块**: `AIRuntimeService`、`SystemStatusService`、`VoiceService`、`SettingsService`
- **目标**: 统一日志格式，减少散落 `print` / `NSLog`
- **任务内容**:
  - 定义日志分类
  - 统一事件/错误/性能/状态输出格式
  - 为关键服务补充文件名、函数名、行号、上下文
  - 把高频调试点收敛到少量 logger sink
- **交付物**:
  - Logger 工具类
  - 替换关键路径输出
  - 日志使用指引
- **预估工期**: 1 - 2 天
- **依赖**: 无
- **验收标准**:
  - 关键服务日志可快速定位问题
  - 不再靠散落打印猜流程

### T03. 系统状态量化与热节流补齐

- **优先级**: P0
- **借鉴来源**: Hot - `pmset -g therm`
- **AcMind 对标模块**: `AcMindKit/Models/SystemStatusModels.swift`、`AcMindKit/Services/SystemStatus/SystemStatusReaders.swift`、`AcMindKit/Services/SystemStatus/SystemStatusService.swift`
- **目标**: 把“热状态”从单一枚举升级为量化信息
- **任务内容**:
  - 新增热节流结构体
  - 读取速度限制、调度限制、可用 CPU 数
  - 在状态页明确显示不可用原因
  - 让系统状态摘要更真实
- **交付物**:
  - 量化热状态模型
  - 新 reader / formatter
  - 测试覆盖
- **预估工期**: 1 - 2 天
- **依赖**: T01 可选
- **验收标准**:
  - 读不到时显示“不可用”而不是假值
  - 读得到时有可解释的量化指标

### T04. 睡眠/唤醒友好采样

- **优先级**: P2
- **借鉴来源**: iGlance - `RepeatingTimer`
- **AcMind 对标模块**: `AcMindKit/Services/SystemStatus/SystemStatusService.swift`
- **目标**: 避免睡眠期间无效轮询，降低无意义 CPU/Timer 开销
- **任务内容**:
  - 把普通 Timer 替换为可 suspend/resume 的调度器
  - 监听 sleep/wake 通知
  - 唤醒后恢复采样节奏
- **交付物**:
  - 睡眠感知定时器封装
  - SystemStatusService 接入
  - 相关测试
- **预估工期**: 0.5 - 1 天
- **依赖**: 无
- **验收标准**:
  - 睡眠期间不继续无效刷新
  - 唤醒后自动恢复

### T05. 用量燃烧率与费用预警

- **优先级**: P1
- **借鉴来源**: cc-statistics - `UsageBurnMonitor`
- **AcMind 对标模块**: `AcMindKit/Services/AI/AIRuntimeService.swift`
- **目标**: 把模型调用从“能用”提升到“可控、可预估”
- **任务内容**:
  - 记录 token、调用量、延迟
  - 计算燃烧率
  - 定义告警阈值和防抖规则
  - 在 UI 给出可理解的风险提示
- **交付物**:
  - 监控模块
  - 告警状态模型
  - UI 提示入口
- **预估工期**: 1 - 2 天
- **依赖**: T02 推荐
- **验收标准**:
  - 能看出“当前消耗是否超速”
  - 告警不会频繁抖动

### T06. 通知焦点检测与降级策略

- **优先级**: P2
- **借鉴来源**: cc-statistics - notifier
- **AcMind 对标模块**: `AcMindKit/Services/Notifications/AppNotificationService.swift`、`AcMindKit/Services/Voice/VoiceNotifications.swift`
- **目标**: 让通知更少打扰，更懂上下文
- **任务内容**:
  - 检测前台焦点
  - 添加系统通知降级路径
  - 重要通知强制发送
  - 在 AcMind 窗口内优先内联提示
- **交付物**:
  - 焦点检测工具
  - 降级通知链路
  - 通知策略说明
- **预估工期**: 0.5 - 1 天
- **依赖**: 无
- **验收标准**:
  - 正在操作时不被过度打断
  - 关键事件不丢

### T07. 本机状态总览页收口

- **交付物**:
