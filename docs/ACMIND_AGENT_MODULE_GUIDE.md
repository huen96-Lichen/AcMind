# AcMind Agent 板块说明

> 面向 ChatGPT 对接与落地实施的 Agent 模块说明
>
> 目标：把 AcMind 的 Agent 板块从“能聊天”升级为“可承接任务、调度能力、回填结果”的产品入口。

## 1. Agent 在 AcMind 里的角色

AcMind 的产品定性是：

- `Agent`：入口
- `日程表`：时间管理
- `工作台`：知识沉淀主流程
- `自动工具`：能力中心
- `设置`：系统规则与配置

其中 `Agent` 不是普通聊天页，而是整个产品的**意图入口**。用户应该首先对 Agent 说“我要做什么”，然后由 Agent 决定：

- 直接回复
- 跳转到工作台某个流程
- 调用自动工具
- 发起整理、检索、导入、导出、任务执行
- 把结果回填到 AcMind 内部对象

### Agent 的核心价值

- 把“用户想做的事”翻译成 AcMind 可执行的动作
- 统一所有入口，不让用户在多个页面之间来回找功能
- 承接自然语言、语音、快捷命令和未来工具调用
- 在不破坏现有工作台/自动工具结构的前提下，提供更自然的交互层

## 2. 当前实现现状

Agent 板块已经不是空壳，它有完整的基础链路：

- 前端页面：`src/renderer/pages/agent-chat/AgentChatPage.tsx`
- 对话逻辑 Hook：`src/renderer/hooks/useChat.ts`
- IPC 暴露：`src/preload/index.ts`
- 主进程处理：`src/main/ipc.ts`
- 对话服务：`src/main/services/chat/chatService.ts`

### 当前已经具备的能力

- 会话创建、切换、删除
- 消息发送与历史加载
- 流式回复
- 停止生成
- 语音输入接入
- 快捷命令
- 基于消息内容的动作建议

### 当前默认行为

Agent 当前默认是：

- `enabled: false`
- `mockMode: true`
- `defaultProviderId: null`
- `defaultModelId: null`

也就是说：

- **没配模型时，Agent 仍然可以运行 UI 和 Mock 对话**
- **不会默认真的调用 ChatGPT 或其他模型**
- **只有配置了 Provider，且关闭 Mock 模式后，才会走真实模型调用**

## 3. 当前用户看到的 Agent

Agent 首页现在承担的是“首屏入口”角色，页面内容包含：

- 会话列表
- 新对话按钮
- 中央输入与对话区
- 快捷命令
- 语音输入状态
- 与工作台、自动工具之间的跳转入口

它已经是一个“可用的 Agent 首页”，但还没有完全变成“真正的任务中枢”。

## 4. Agent 的产品边界

Agent 应该做什么：

- 接收自然语言任务
- 判断任务属于哪个模块
- 发起导航或调用工具
- 读取必要上下文
- 汇总结果并回写

Agent 不应该做什么：

- 代替工作台存储知识
- 代替自动工具承载所有具体能力
- 变成一个无限膨胀的控制台
- 把所有页面都塞进聊天窗口

### 简单理解

- `Agent` 负责“理解和调度”
- `工作台` 负责“收集、整理、确认、入库”
- `自动工具` 负责“执行具体能力”
- `设置` 负责“配置模型、路径、权限和行为”

## 6. 现在还缺什么

如果要把 Agent 真正和 ChatGPT 对接落地，当前还缺少几类“产品级能力”：

### 6.1 真实模型配置闭环

目前 Agent 依赖 `Provider` 配置。要让它稳定接入 ChatGPT，需要明确：

- 选择哪一种 Provider 作为默认对接方式
- 使用 OpenAI-compatible 接口还是官方接口
- API Key 存储在哪里
- 默认模型如何选择
- 失败回退策略是什么

### 6.2 意图到动作的映射

现在 Agent 可以发消息和给动作建议，但还需要更明确的任务分流：

- 什么情况直接答复
- 什么情况跳工作台
- 什么情况跳自动工具
- 什么情况生成待办
- 什么情况发起导出

### 6.3 上下文注入

Agent 要真正好用，必须知道“用户现在在看什么”：

- 当前工作台条目
- 当前暂存内容
- 当前知识卡片
- 当前任务上下文
- 当前页面的选中对象

### 6.4 工具调用协议

如果要和 ChatGPT 对接，建议把工具调用收敛成统一协议：

- 查询类工具
- 导航类工具
- 任务类工具
- 写回类工具
- 外部执行类工具

## 7. 推荐的落地方式

### 方式 A：先做纯对话 Agent

适合快速上线。

流程：

1. 用户输入问题
2. Agent 通过 ChatGPT 回复
3. 如果需要动作，返回结构化建议
4. 用户手动确认

优点：

- 实现快
- 风险低
- 容易调试

缺点：

- 还不够“Agent”
- 动作闭环弱

### 方式 B：对话 + 工具调用

推荐作为正式目标。

流程：

1. 用户输入任务
2. ChatGPT 分析任务意图
3. 根据需要调用 AcMind 内部工具
4. 把结果回写到工作台或自动工具
5. 形成可追踪的任务链

优点：

- 更符合“Agent-first”
- 更接近真正落地

缺点：

- 需要清晰的工具协议
- 需要更多权限与错误控制

## 9. 与 ChatGPT 对接时的建议

如果你准备让 ChatGPT 真正参与 AcMind 的 Agent 板块，建议采用下面的原则：

### 9.1 ChatGPT 负责“理解”，AcMind 负责“执行”

ChatGPT 不直接控制所有业务细节，而是：

- 理解用户意图
- 选择工具
- 生成可执行结构
- 处理自然语言输出

AcMind 负责：

- 数据读写
- 页面跳转
- 文件和知识库操作
- 任务状态管理

### 9.2 所有动作都要结构化

不要让模型只返回一段自然语言就结束，最好统一成：

- `answer`
- `suggested_actions`
- `requires_confirmation`
- `target_view`
- `payload`

### 9.3 保持产品边界清晰

不要把 Agent 做成“啥都能问的聊天窗”。

它应该是：

- 产品入口
- 任务调度器
- 上下文解释器
- 能力分发器

不是：

- 一个孤立的 LLM Demo
- 一个新聊天软件
- 一个控制台后台

## 10. 建议给 ChatGPT 的任务描述

你可以把下面这段作为对接时的任务背景：

> AcMind 是一款 Agent-first 的个人桌面 AI 管家。  
> 其中 Agent 是默认首页和意图入口，负责接收用户自然语言任务，并把任务分发到工作台、自动工具或设置中的相关能力。  
> 当前 Agent 已具备会话、消息、流式输出和动作建议能力，但还需要接入真实模型并建立工具调用与结果回填闭环。  
> 你的任务不是重写整个产品，而是帮助 AcMind 的 Agent 变成一个真正能理解任务、调用工具、回写结果的入口层。

## 11. 现阶段结论

Agent 板块现在的状态可以概括为：

- **页面和 IPC 是活的**
- **Mock 对话是可用的**
- **真实模型接入是预留好的**
- **真正的 ChatGPT 落地还差“模型配置 + 工具调用 + 回填闭环”**

如果要真正完成落地，下一步应该优先把：

1. 模型配置打通
2. 工具调用协议定下来
3. 结果回填到工作台/自动工具
4. 再做动作自动化和任务编排
