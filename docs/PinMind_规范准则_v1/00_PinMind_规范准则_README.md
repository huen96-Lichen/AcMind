# AcMind 规范准则文件夹 README v1

## 1. 文件夹用途

本文件夹用于存放 AcMind 项目的高优先级规范准则。

它不是普通参考资料，而是项目推进过程中的“共同规则层”。

适用对象：

- 产品负责人：用于判断方向是否跑偏
- ChatGPT：用于制定规划、PRD、验收标准
- Trae：用于大体代码落地
- Codex：用于核验、审计、精修
- 未来其他设备 / 其他开发环境：用于保持一致的项目判断标准

## 2. AcMind 的项目定位

AcMind 不是 PinStack 的简单改名。

AcMind 的核心定位是：

> AcMind 是一个本地优先的 AI 第二大脑蒸馏器。  
> 它通过本地模型和可控的云端模型，自动捕获、筛选、整理、清理和输出你的个人信息，让碎片内容变成可搜索、可问答、可沉淀到 Obsidian 的长期记忆。

## 3. 项目主链路

所有功能必须围绕这条链路展开：

```txt
捕获信息
↓
理解信息
↓
蒸馏信息
↓
归档信息
↓
复用信息
```

对应到实际产品行为：

```txt
复制 / 输入 / 导入 / 截图 / 音视频转写
↓
进入本地 Inbox
↓
AI 识别、清理、总结、打标签
↓
生成稳定 Markdown
↓
写入 iCloud 同步的 Obsidian Vault
↓
未来可以搜索、问答、复盘、复用
```

## 4. 最高优先级原则

1. 先闭环，再扩展。
2. 先本地可用，再云端增强。
3. 先 Mock 跑通，再接真实模型。
4. 先 Markdown 输出稳定，再做知识图谱。
5. 先 Obsidian 沉淀，再做多端同步。
6. UI 必须接真实状态。
7. 设置必须持久化。
8. AI 输出必须可预览、可撤销。
9. 每个版本必须 Codex 核验。
10. 任何偏离“第二大脑蒸馏器”的功能都进入 backlog。

## 6. 使用方式

每次新任务开始前：

1. 先看 `01_项目定位与边界规范.md`
2. 再看和任务相关的专项规范
3. 给 Trae 的任务必须带上验收标准
4. Trae 完成后必须写交付报告
5. Codex 必须按 `09_Codex核验与质量门禁规范.md` 核验
6. 没有通过核验，不进入下一版本

## 7. 文件维护规则

本文件夹里的规范属于项目高优先级规则。

修改规范必须遵守：

- 不能随手改
- 修改必须写明原因
- 修改后要同步给 Trae 和 Codex
- 重大变更要更新版本号
- 如果规范与临时代码实现冲突，以规范为准，除非你明确决定修改规范
