# PinMind 三方协作工作流规范 v1

## 1. 协作角色

PinMind 当前采用四方协作：

- 产品负责人：你
- 产品经理 / 架构约束者：ChatGPT
- 主力实现者：Trae
- 核验与精修者：Codex

## 2. 产品负责人职责

你负责：

- 决定产品方向
- 判断体验是否符合预期
- 决定功能优先级
- 决定是否进入下一版本
- 决定是否接受技术债
- 决定是否修改规范

你不需要每天深入所有代码细节，但必须控制方向。

## 3. ChatGPT 职责

ChatGPT 负责：

- 大方向规划
- 产品手册
- PRD
- 技术约束
- UI / UX 规范
- 版本路线图
- 任务拆解
- 验收标准
- 风险提示
- 判断项目是否跑偏

ChatGPT 的输出应该尽量是文档化、可交付、可复制的内容。

## 4. Trae 职责

Trae 负责约 95% 代码落地，包括：

- 页面实现
- 组件实现
- 状态管理
- 本地存储
- IPC 接入
- 常规服务层
- 基础重构
- UI 落地
- 常规 bug 修复

Trae 不负责最终决定：

- 产品定位
- 主数据模型
- AI 调用策略
- Obsidian 输出规范
- 版本节奏
- 是否引入重大技术栈
- 是否新增大模块

## 5. Codex 职责

Codex 负责：

- 只读核验
- 精准修复
- 类型检查
- 架构检查
- 持久化检查
- 主链路检查
- 假功能识别
- 性能问题识别
- 小范围高质量精修

Codex 不是第二个 Trae。

Codex 的重点不是写很多代码，而是确认 Trae 写的代码是否真的可靠。

## 6. 标准推进流程

```txt
你提出方向
↓
ChatGPT 输出 PRD / 技术约束 / 验收清单
↓
Trae 根据文档实现
↓
Trae 输出交付报告
↓
Codex 进行只读核验
↓
ChatGPT 帮你判断是否通过
↓
Trae 按核验意见修复
↓
Codex 二次核验
↓
版本封版
```

## 7. 禁止跳过的环节

禁止：

- 没有 PRD 就让 Trae 开写
- 没有验收标准就认为完成
- 没有 Codex 核验就进入下一版本
- Trae 说完成就直接相信
- UI 做完但主链路未接入
- Mock 功能伪装成真实功能
- 临时需求直接写进主线

## 8. 每次任务的沟通格式

给 Trae 的任务必须包含：

```md
# PinMind vX.X.X 任务说明

## 一、任务目标

## 二、本次必须完成

## 三、本次禁止完成

## 四、涉及文件范围

## 五、数据结构要求

## 六、UI 要求

## 七、交互要求

## 八、错误处理要求

## 九、日志要求

## 十、验收标准

## 十一、完成后必须更新的文档

## 十二、交付报告格式
```

## 9. Trae 交付报告格式

Trae 每次完成后必须输出：

```md
## 本次完成

## 修改文件

## 新增文件

## 删除文件

## 运行方式

## 验收方式

## 已知问题

## 未完成内容

## 需要 Codex 重点核验的地方
```

没有“已知问题”和“未完成内容”的交付报告，视为不完整。

## 10. Codex 核验报告格式

Codex 每次核验必须输出：

```md
## 结论
通过 / 部分通过 / 不通过

## 真实完成项

## 假完成项

## 主要风险

## 必须修复

## 建议修复

## 是否允许进入下一版本
是 / 否
```
