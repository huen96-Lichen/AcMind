# AcMind 需求拆解与任务交付规范 v1

## 1. 为什么需要这个规范

Trae 是主力实现者，但不能给它模糊任务。

模糊任务会导致：

- UI 做了但逻辑没接
- 功能看起来有但实际不可用
- 设置没有持久化
- Mock 被当成真实功能
- 开发范围被自动扩大
- Codex 核验时大量返工

所以每个任务必须拆得足够清楚。

## 2. 不合格任务示例

```txt
帮我优化 AI 面板
```

问题：

- 不知道优化什么
- 不知道要改哪些文件
- 不知道验收标准
- 不知道是否要接真实模型
- 不知道是否要持久化
- 不知道是否要写日志

## 3. 合格任务示例

```md
# AcMind v0.3 AI 任务队列 Mock 版

## 任务目标
跑通 SourceItem 从 Inbox 发起 AI 蒸馏任务的完整链路。

## 必须完成
1. 新增 AiTask 类型
2. 支持 pending / running / success / failed
3. InboxItem 可以发起 summarize 任务
4. 任务结果保存到本地
5. UI 显示任务进度
6. 失败任务可以重试
7. 重启后任务记录仍存在
8. 更新 CHANGELOG 和 PROJECT_HANDOVER

## 禁止完成
1. 不接真实云端模型
2. 不引入账号系统
3. 不做模型市场
4. 不做复杂知识图谱

## 验收标准
1. 新建 SourceItem 后可以发起 AI 任务
2. 任务状态会变化
3. 任务结果可查看
4. 重启后任务记录仍在
5. 失败有错误提示
```

## 4. 每个任务必须包含的内容

每个任务至少包含：

- 任务目标
- 必须完成
- 明确禁止
- 涉及文件范围
- 数据结构要求
- UI 要求
- 交互要求
- 错误处理
- 日志要求
- 验收标准
- 文档更新要求
- 交付报告格式

## 5. 任务粒度约束

一个任务最好控制在：

- 1 个主目标
- 1 条主链路
- 1 到 3 个主要页面或模块
- 3 到 8 个主要验收点

如果超过这个范围，应该拆成多个任务。

## 6. 明确禁止项非常重要

每次任务都要写“本次禁止完成”。

原因：

- 防止 Trae 自由发挥
- 防止版本范围膨胀
- 防止主线被外围功能污染
- 防止代码结构提前复杂化

## 7. 交付后必须回答的问题

Trae 每次交付后必须回答：

1. 本次改了什么？
2. 哪些文件被修改？
3. 哪些文件是新增？
4. 如何运行？
5. 如何验收？
6. 哪些地方没完成？
7. 哪些地方可能有风险？
8. 需要 Codex 重点看哪里？

## 8. 不允许口头完成

不能只说：

```txt
已经完成了
```

必须提供结构化交付报告。

## 9. 任务状态

建议每个任务文档记录状态：

```md
status: draft / ready / in_progress / delivered / verified / sealed / rejected
```

含义：

- draft：草稿
- ready：可以交给 Trae
- in_progress：Trae 正在做
- delivered：Trae 已交付
- verified：Codex 已核验通过
- sealed：版本封版
- rejected：不通过，需要重做或废弃
