# PinMind Codex 核验与质量门禁规范 v1

## 1. Codex 的定位

Codex 是 PinMind 项目的：

- 验收官
- 代码审计者
- 精修工程师
- 假功能识别者
- 质量门禁

Codex 不应该成为第二个 Trae。

## 2. 核验原则

Codex 不看“Trae 说完成了”，只看代码事实。

必须确认：

- 类型是否真实存在
- 逻辑是否真实接入
- 设置是否真实持久化
- UI 是否真实消费状态
- 数据是否真实保存
- 重启后是否仍然有效
- 错误是否真实反馈
- 日志是否真实写入


## 4. 必查项目

### 4.1 类型检查

检查：

- interface 是否完整
- type 是否被使用
- 是否存在重复定义
- 是否存在 any 滥用
- 是否存在 UI 层私自定义核心类型
- 是否与主数据模型冲突

### 4.2 持久化检查

检查：

- UI 设置是否写入主配置
- 主进程是否读取
- 重启后是否仍在
- 是否只是存在 React state
- 是否保存到正确目录
- 错误时是否有 fallback

### 4.3 主链路检查

检查：

- 按钮是否真的触发逻辑
- 状态是否真的变化
- 数据是否真的保存
- 失败是否能重试
- 成功是否进入下一阶段
- 是否绕开 SourceItem 主模型

### 4.4 UI 检查

检查：

- 是否可滚动
- 是否信息被遮挡
- 是否有不可点击的假按钮
- 是否所有点击都有反馈
- 是否有 loading / error / empty 状态
- 是否小窗口下可用

### 4.5 Mock 检查

检查：

- Mock 是否明确标注
- 是否伪装成真实模型
- 是否影响真实 provider 接入
- 是否有切换机制
- 是否能用于链路演示

### 4.6 回归检查

检查：

- 应用能否启动
- 主窗口是否正常
- 设置是否正常
- 旧数据是否可读
- 日志是否正常
- 构建是否通过
- 类型检查是否通过

## 5. 不允许通过的情况

出现以下情况不能通过：

- UI 有按钮但无逻辑
- 设置不能持久化
- 重启后数据丢失
- Mock 冒充真实功能
- 主链路被绕开
- 数据直接写进源码目录
- 错误无提示
- 日志无记录
- 页面不能滚动导致信息被遮挡
- 版本文档未更新

## 6. 质量门禁等级

### P0 必须立刻修

- 应用无法启动
- 数据丢失
- 设置丢失
- 主链路不可用
- Obsidian 输出覆盖旧文件
- AI 输出覆盖原文
- 页面严重不可用

### P1 当前版本必须修

- 交互无反馈
- 日志缺失
- 错误提示不清楚
- 状态不完整
- 文档未更新
- UI 可读性问题

### P2 可进入后续优化

- 小型样式问题
- 文案优化
- 组件复用不足
- 局部代码可读性一般
- 非主链路性能优化

## 7. 是否允许进入下一版本

Codex 必须明确给出：

```txt
是否允许进入下一版本：是 / 否
```

如果是“部分通过”，默认不能进入下一版本，除非产品负责人明确接受剩余风险。
