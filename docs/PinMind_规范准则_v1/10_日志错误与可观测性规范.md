# PinMind 日志错误与可观测性规范 v1

## 1. 设计目标

PinMind 必须避免“点了没反应”。

所有关键动作都应该被用户看见，被系统记录，被开发者追踪。

## 2. 必须记录日志的动作

至少包括：

- 应用启动
- 应用退出
- 设置读取
- 设置保存
- 捕获内容
- 保存 SourceItem
- 创建 AI 任务
- AI 任务开始
- AI 任务成功
- AI 任务失败
- AI 结果保存
- Markdown 生成
- Obsidian 输出
- Provider 检测
- Provider 不可用
- 文件读写失败
- 未捕获异常

## 3. 推荐日志文件

```txt
logs/
├── app.log
├── ai.log
├── export.log
└── error.log
```

## 4. 日志内容

每条日志建议包含：

- 时间
- 模块
- 动作
- 状态
- 关联 id
- 用户可读信息
- 技术细节

示例：

```json
{
  "time": "2026-04-27T15:00:00+09:00",
  "module": "ai",
  "action": "run_task",
  "status": "failed",
  "taskId": "task_123",
  "sourceId": "source_456",
  "message": "AI 处理失败：云端模型 API Key 未配置",
  "detail": "Missing API key for cloud_standard provider"
}
```

## 5. 用户错误提示原则

用户提示必须说明：

- 发生了什么
- 为什么可能发生
- 用户能做什么
- 是否可以重试

错误示例：

```txt
Error: undefined
```

正确示例：

```txt
AI 处理失败：当前云端模型 API Key 未配置，请前往设置页补充。
```

## 6. UI 状态

所有异步动作至少需要：

- idle
- loading
- success
- error

AI 任务需要：

- pending
- running
- success
- failed
- cancelled

## 7. 重试机制

以下动作需要支持重试：

- AI 任务
- Markdown 导出
- Provider 检测
- 文件读取
- 文件写入

重试失败后必须保留错误信息。

## 8. 崩溃保护

应用不应因为单个任务失败而崩溃。

必须保证：

- 单个 AI 任务失败不影响其他任务
- 单个文件导出失败不影响其他数据
- Provider 不可用不影响本地 Inbox
- API Key 缺失不影响应用启动

## 9. 可观测性页面

后续建议提供“运行状态 / 日志中心”页面，显示：

- 最近任务
- 最近错误
- Provider 状态
- 数据目录状态
- Obsidian 路径状态
- 日志文件入口

早期可以先做基础日志文件，不必立即做完整 UI。
