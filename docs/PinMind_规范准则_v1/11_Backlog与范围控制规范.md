# AcMind Backlog 与范围控制规范 v1

## 1. 为什么需要 Backlog

AcMind 很容易扩展出大量方向：

- 截图
- OCR
- 视频理解
- 语音转写
- 网页剪藏
- 知识图谱
- AI 控制台
- 顶部胶囊
- 多设备同步
- 本地模型管理
- 云端模型管理
- 个人主页
- NAS 集成

这些方向都有价值，但不能同时进入主线。

Backlog 的作用是：

> 先保存想法，但不打断当前版本。

## 2. Backlog 文件

建议建立：

```txt
docs/IDEA_BACKLOG.md
```

## 3. Backlog 条目格式

```md
## 想法名称

### 来源
这个想法从哪里来？

### 价值
它解决什么问题？

### 所属主链路
捕获 / 理解 / 蒸馏 / 归档 / 复用 / 外围增强

### 优先级
P0 / P1 / P2 / P3

### 为什么现在不做
说明暂缓原因。

### 进入条件
未来满足什么条件后可以开发？
```

## 4. 优先级定义

### P0 当前主线必须做

不做就无法完成核心闭环。

例如：

- 本地 Inbox
- AI Mock 蒸馏
- Obsidian 输出
- 设置持久化

### P1 近期增强

核心闭环跑通后，很快应该做。

例如：

- 真实 AI Provider
- AI 控制面板
- 批量处理
- Markdown 规范增强

### P2 中期能力

产品稳定后再做。

例如：

- OCR
- 网页剪藏
- 音频转写
- 视频文案提取
- 本地搜索

### P3 长期探索

未来可能做，但现在不能影响主线。

例如：

- 插件市场
- 多端同步
- 团队协作
- 复杂知识图谱
- 商业化面板

## 5. 禁止直接进入开发的功能

以下功能必须先进入 Backlog，不能直接开发：

- 账号系统
- 插件市场
- 团队协作
- 云同步平台
- 社交分享
- 商城
- 复杂权限系统
- 重型后端服务
- 非必要网页端
- 多端 App

## 6. 判断是否进入当前版本

一个 Backlog 想法进入当前版本前，需要回答：

1. 是否服务当前版本唯一目标？
2. 是否服务主链路？
3. 是否会带来明显复杂度？
4. 是否会影响已有闭环？
5. 是否有明确验收标准？
6. 是否能被 Codex 核验？

如果不能全部回答清楚，不进入当前版本。

## 7. 范围失控警报

出现以下情况，说明项目正在失控：

- 一个版本同时做 5 个以上大功能
- Trae 自行新增大模块
- UI 入口越来越多但主链路没跑通
- 设置越来越多但无法持久化
- 文档跟不上代码
- Codex 核验发现大量假功能
- 用户无法一句话说清当前版本目标

一旦出现这些情况，应暂停新功能，先做收敛。
