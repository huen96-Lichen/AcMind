# AcMind 默认输出与 Format Profile 规范 v0.2

## 0. 文档定位

本文档用于规定 AcMind 默认输出内容的结构、字段、模板、维护方式和迁移规则。

它的目标是让以下对象都能快速理解并长期使用：

- 普通用户
- 产品设计者
- 开发者
- 本地大模型
- 云端大模型
- 后续自动化脚本
- Obsidian 知识库维护流程

本文档的核心原则是：

> AcMind 的默认输出系统采用“三层分离”设计：  
> **标准字段层负责语义，Format Profile 层负责渲染，Markdown 文件层负责长期存储。**  
> 模型永远只输出标准字段，模板永远只处理字段渲染，迁移永远只改变格式，不改变语义。

---

## 1. AcMind 默认体验目标

AcMind 的默认体验应该尽量简单。

用户只需要负责：

- 收集内容
- 查看结果
- 必要时手动修正

系统负责：

- 清洗内容
- 结构化内容
- 生成标题
- 生成总结
- 生成标签
- 判断分类
- 套用统一格式
- 导出到 Obsidian 或 Markdown 目录

最终用户在 Obsidian 中看到的内容应该是：

- 格式统一
- 一眼能看懂
- 可以手动编辑
- 可以批量维护
- 可以长期迁移
- 即使没有 AI，也能继续使用

---

## 2. 设计原则

### 2.1 默认只提供一套推荐格式

AcMind 默认只提供一套稳定的 Markdown 输出格式。

原因：

- 降低用户选择成本
- 降低模型输出成本
- 降低维护复杂度
- 降低后续迁移风险

高级用户可以在设置中创建多个 `Format Profile`，但普通用户不需要理解这些内容。

---

### 2.2 内容语义和文档格式分离

AcMind 内部不应该让模型直接输出完整 Markdown。

模型只负责输出结构化字段。

最终 Markdown 由系统根据模板生成。

这样可以保证：

- 模型容易理解
- 输出更稳定
- 模板可以随时更换
- 老内容可以批量重新渲染
- 用户不用逐条手动改格式

---

### 2.3 用户必须可以脱离 AI 手动维护

AcMind 输出的 Markdown 不能变成只有模型才能理解的格式。

即使没有 AI，用户也应该能够：

- 看懂标题
- 看懂总结
- 修改标签
- 修改分类
- 修改正文
- 修改项目归属
- 重新移动文件
- 批量搜索和替换
- 在 Obsidian 中正常维护

所以默认输出必须保持人类可读，而不是只服务于机器。

---

### 2.4 所有字段都应该稳定

字段一旦确定，就不应该频繁改名。

字段稳定的好处：

- 方便搜索
- 方便筛选
- 方便批量迁移
- 方便写脚本
- 方便训练模型
- 方便长期维护知识库

---

## 3. 三层结构

AcMind 的输出系统分为三层。

```text
标准字段层
↓
Format Profile 渲染层
↓
Markdown 文件层
```

### 3.1 标准字段层

标准字段层负责“这段内容是什么”。

它描述内容的语义，不关心 Markdown 长什么样。

这一层主要服务于：

- 模型
- 数据库
- 自动化脚本
- 迁移系统
- 搜索系统

### 3.2 Format Profile 渲染层

Format Profile 负责“如何把标准字段渲染成 Markdown”。

它只处理格式，不处理语义。

### 3.3 Markdown 文件层

Markdown 文件层是用户最终看到和维护的结果。

这一层必须满足：

- Obsidian 友好
- 人类可读
- 可手动编辑
- 可批量处理
- 可长期保存
- 不依赖 AcMind 也能打开

---

## 4. 标准字段总览

AcMind 内部建议使用英文 key，用户展示时使用中文名称。

这样可以同时兼顾：

- 工程稳定性
- 模型理解
- 用户可读性
- Obsidian 维护体验

| 中文名称 | 内部 key | 类型 | 是否必填 | 默认值 | 说明 |
|---|---|---:|---:|---|---|
| 标题 | `title` | string | 是 | 无 | 文档主名称 |
| 总结 | `summary` | string | 是 | 无 | 一句话到两句话总结 |
| 标签 | `tags` | string[] | 否 | `[]` | 用于检索和聚合 |
| 分类 | `category` | string | 否 | `未分类` | 主归类，只保留一个 |
| 正文 | `body` | string | 是 | 无 | 整理后的正文 |
| 原始内容 | `raw_content` | string | 否 | 空 | 原始输入内容，可选保留 |
| 来源 | `source` | string | 是 | `manual` | 内容来源 |
| 采集时间 | `captured_at` | string | 是 | 当前时间 | 内容采集时间 |
| 项目 | `project` | string | 否 | `默认` | 所属项目 |
| 状态 | `status` | string | 是 | `collected` | 当前流程状态 |
| 置信度 | `confidence` | number | 否 | `0.5` | 模型处理可信度 |
| 格式版本 | `schema_version` | string | 是 | `0.2` | 当前字段规范版本 |

## 5. 默认 Markdown 输出结构

AcMind 默认输出结构为：

```text
frontmatter 属性区
↓
一句话总结
↓
正文标题
↓
正文内容
↓
可选原始内容
```

推荐默认模板：

```markdown
---
schema_version: "0.2"
title: "{{title}}"
summary: "{{summary}}"
tags: {{tags}}
category: "{{category}}"
source: "{{source}}"
captured_at: "{{captured_at}}"
project: "{{project}}"
status: "{{status}}"
confidence: {{confidence}}
---

> {{summary}}

# {{title}}

{{body}}
```

---

## 6. 模型输出规则

模型不应该直接输出完整 Markdown。

模型应该输出 JSON 或等价结构化字段。

推荐格式：

```json
{
  "schema_version": "0.2",
  "title": "AcMind 默认输出规范",
  "summary": "本文定义 AcMind 如何把收集内容转化为统一、可维护的 Markdown 文档。",
  "tags": ["AcMind", "Markdown", "Obsidian"],
  "category": "产品规范",
  "body": "这里是整理后的正文内容。",
  "raw_content": "这里是原始输入内容。",
  "source": "manual",
  "captured_at": "2026-04-30 10:34",
  "project": "AcMind",
  "status": "cleaned",
  "confidence": 0.92
}
```

模型必须遵守：

1. 不编造来源
2. 不编造时间
3. 不强行分类
4. 不制造过多标签
5. 不删除关键信息
6. 不把总结写成长文
7. 不把正文压缩成摘要
8. 不改变用户原始意图
9. 不把格式规则和内容语义混在一起

---

## 7. Format Profile 规则

Format Profile 是 AcMind 用来控制 Markdown 输出样式的一套配置。

它只负责“如何渲染”，不负责“内容是什么”。

可以配置：

- 文件命名规则
- frontmatter 字段映射
- 标签格式
- 日期格式
- 标题层级
- 总结显示方式
- 正文拼接方式
- 原始内容是否展示
- 属性区字段顺序
- 导出路径规则

不应该配置：

- 模型理解
- 内容总结
- 内容分类
- 标签生成
- 项目判断
- 来源判断
- 置信度计算
- 状态流转逻辑

---

## 8. 文件命名规则

推荐默认文件名：

```text
YYYY-MM-DD HH-mm - 标题.md
```

示例：

```text
2026-04-30 10-34 - AcMind 默认输出规范.md
```

文件名应该：

- 可读
- 可排序
- 不过长
- 不包含特殊危险字符
- 尽量和标题一致

生成文件名时应自动替换以下字符：

```text
/ \ : * ? " < > |
```

---

## 9. 批量迁移规则

AcMind 应支持三种迁移范围：

1. 仅影响新内容
2. 影响整个 vault
3. 只影响某个 project

迁移时只做：

1. 重新渲染模板
2. 保留标准字段不变

迁移不应该：

- 改变标题语义
- 改变总结内容
- 改变正文内容
- 自动删除用户手写内容
- 随意重命名标签
- 随意改变分类
- 覆盖原始内容
- 让用户逐条手动修格式

批量迁移前必须：

- 生成备份
- 生成迁移日志
- 允许用户预览
- 允许用户回滚

---

## 10. 人工维护规则

即使没有 AI，用户也可以手动维护 AcMind 输出的 Markdown。

用户可以直接修改：

- 标题
- 总结
- 标签
- 分类
- 项目
- 正文
- 状态
- 置信度

如果修改标题，建议同步修改：

- frontmatter 里的 `title`
- 正文一级标题 `# 标题`
- 文件名

分类只保留一个主分类。复杂关系交给标签。

状态建议使用固定英文枚举。

---

## 11. 最终结论

AcMind 默认输出规范的核心不是让文档变复杂，而是让内容变得：

- 一眼能看懂
- 长期能维护
- 模型能处理
- 用户能手改
- Obsidian 能承接
- 未来能批量迁移

标准字段负责语义，Format Profile 负责格式，Markdown 文件负责长期保存。

只要这三层不混在一起，AcMind 的内容系统就可以长期稳定演进。
