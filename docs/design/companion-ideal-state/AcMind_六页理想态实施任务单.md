# AcMind 六页理想态实施任务单

> 这是一份给 coding agent 的执行单，不是讨论稿。  
> 目标只有一个，把六个界面收敛到同一套壳层、同一套节奏、同一套控件语法里，且不引入新的产品路径。
> 
> 最终容器固定为 880 × 300，所有页面的实际实现和导出的参考图都必须严格遵守这个尺寸。

## 一句话目标

把 AcMind 的六个 companion 页面改成一套可以长期维护的、统一的 macOS 工具界面。  
它们可以承载不同内容，但不能再长得像六个不同产品。

## 交付标准

完成后，必须满足以下条件：

- 六个页面共享同一套外壳、顶栏、底栏、三列比例和卡片语法。
- 六个页面的视觉层级一致，用户一眼能识别主内容、次级内容、辅助状态。
- 所有关键状态都能在首屏内读到，不依赖滚动。
- 所有页面的控件密度是高的，但不是乱的。
- 代码可以编译通过，且没有引入新的导航逻辑或新的数据来源。

### 可感知结果门槛

为了保证你起床后能“明显感知”到改动，而不是只看到一些边角调色，最终交付必须满足这些可感知标准：

- 每个页面最多只有 1 个主动作。
- 每个页面首屏最多同时暴露 3 个高优先级次动作。
- 页面状态变化要有立即可见反馈，不能只改数据不改样式。
- 按钮 / toggle / selected 状态的视觉反馈应在 150 到 250 ms 内完成。
- 空态、加载态、处理中状态必须有明确文字，不允许只靠转圈。
- 同一页里重复解释同一状态的文案，最多保留一处。

## 不做什么

这个任务单明确排除以下方向：

- 不新增页面
- 不新增导航层级
- 不改数据采集逻辑
- 不改真实能力来源
- 不做装饰性动效
- 不做玻璃拟态
- 不让每页再发明一套视觉语言

## 架构边界先决

这次不是单纯改 token，也不是只改页面内容，先把 companion 的显示边界说清楚。

- 折叠态是菜单栏上的最小状态条。
- 展开态是 880 × 300 的主面板。
- 这份任务单里的固定规格，只针对展开态主面板。
- 折叠态只需要保持现有收起行为和状态入口，不需要跟展开态共用同一套像素尺寸。

## 核心设计原则

### 1. 壳层固定，内容可变

所有页面必须共享同一套：

- 顶栏
- 底栏
- 左中右布局骨架
- 页面内卡片圆角
- 阴影强度
- 边框分隔方式

页面能变化的只有内容，不是容器语法。

### 2. 单页 source of truth 规则

如果一个页面在代码里同时存在 `NotchV2*Page` 和 `DynamicContinent*Page` 两个版本，不允许各自独立演化。

- `DynamicContinent*Page` 是展开态的运行入口和视觉基线。
- `NotchV2*Page` 只能作为兼容包装或共享子视图来源。
- 任何视觉规范变更必须先落到共享子视图，再让两个版本一起对齐。

### 3. 主任务优先

每页只能有一个主任务。

如果一页里出现两个主任务，就说明层级没收住。

### 4. 先结论，后细项

所有页面先给结论，再给细节。

例如：

- 本机页先看当前任务，再看快捷动作和系统快览
- 启动器页先看搜索，再看常用应用和全部应用
- 音乐页先看播放状态，再看队列和控制
- AI 页先看对话，再看历史和就绪状态
- 系统状态页先看健康结论，再看指标和权限
- 设置页先看总开关，再看布局和模块

### 5. 控件语气统一

所有按钮、胶囊、菜单、toggle、slider 的语气必须一致。

同一类操作不能在不同页面里换一种样子。

## 页面级任务

### 0.5 统一壳层范围说明

- `Schedule` 页不单独出现在这份六页理想态的视觉主图里，但它必须遵守同一套展开态壳层、圆角、字号和列宽语法。
- `Schedule` 页不做新的视觉分叉，只做壳层对齐和卡片语法收敛。
- 如果执行时发现 `Schedule` 页需要额外 token，只允许补共享 token，不允许单独长出一套自己的窗口语法。

### 任务 1，本机页

目标：让首页像一个总览仪表盘，而不是一个功能拼盘。

保留：

- 当前任务
- 运行摘要
- 快捷动作
- 系统快览

删除或收窄：

- 次级说明句
- 重复状态标签
- 不产生动作价值的按钮

布局要求：

- 左列只讲当前在做什么
- 中列只放最常用的 2 到 3 个动作
- 右列只讲机器状态是否正常

验收：

- 用户进入后 2 秒内能看懂当前任务是什么
- 中列动作数不多于 3 行
- 右列状态不再像报表

### 任务 2，启动器页

目标：让启动器像搜索工具，不像分类目录。

保留：

- 搜索框
- 常用应用
- 全部应用

删除或收窄：

- 多余说明
- 多余按钮
- 和搜索无关的控制噪声

布局要求：

- 搜索框放在视觉重心
- 常用应用是第一视觉焦点
- 全部应用是结果列表，不是第二个主视觉

验收：

- 用户一眼知道先搜再点
- 常用应用区与全部应用区的主次清楚
- 搜索结果不会把页面挤成杂乱目录

### 任务 3，音乐页

目标：让音乐页像成熟播放器，而不是一组媒体控件。

保留：

- 播放列表
- 正在播放
- 播放控制

删除或收窄：

- 空态长句
- 重复辅助区块
- 说明性质的附加文案

布局要求：

- 左列负责上下文
- 中列负责当前播放
- 右列负责控制和状态

验收：

- 没播放时页面依然完整
- 播放时主视觉聚焦在当前歌曲
- 控制区不抢播放内容的注意力

### 任务 4，AI 页

目标：让 AI 页像对话工作台，不像聊天玩具。

保留：

- 对话历史
- 对话主区
- 快速提问
- 输入框
- 发送动作

删除或收窄：

- 额外状态卡
- 过多快捷指令
- 装饰性“就绪”块

布局要求：

- 左侧是历史，不是导航
- 中间是输入和对话，不是海报
- 右侧只保留最轻状态

验收：

- 用户能马上开始问问题
- 输入框是唯一主动作
- 历史区不抢主区注意力

### 任务 5，系统状态页

目标：让系统状态页像监控面板，而不是性能报表。

保留：

- 运行摘要
- 核心摘要
- 本机状态
- 权限状态

删除或收窄：

- 重复总结语
- 过多解释句
- 过多小卡堆叠

布局要求：

- 左列放机器基础信息
- 中列先给健康结论，再给 4 到 6 个关键指标
- 右列放设备状态和权限状态

验收：

- 先看到结论，再看到数据
- 指标数量控制在 4 到 6 个核心项
- 权限区是辅助，不是主视觉

### 任务 6，设置页

目标：让设置页像专业配置面板，而不是实验室控制台。

保留：

- 展开与行为
- 页面布局
- 子模块管理

删除或收窄：

- 每行重复说明
- 并列过多的小控件
- 视觉噪声过强的控制样式

布局要求：

- 左列是运行行为
- 中列是布局参数
- 右列是模块排序与显隐

验收：

- 用户能快速找到某个设置
- 设置行的语气统一
- 模块管理仍然密，但不乱

## 实施顺序

按下面顺序做，不要反过来：

### Phase 1，壳层统一

先锁：

- 顶栏高度
- 底栏高度
- 左中右列宽
- 卡片圆角
- 阴影和边框

### Phase 2，页面主次统一

再收：

- 页面标题
- 副标题
- 说明句
- 额外按钮

### Phase 3，页面内部密度整理

最后调：

- 各页内部的模块顺序
- 列表高度
- 空态文案
- 控件大小

## coding agent 执行规则

给执行者的硬规则如下：

- 只在指定文件内改动，别扩散到无关模块。
- 每改完一组页面就编译一次。
- 不要在没有必要的时候改数据层。
- 不要重新发明组件，只在现有组件上收敛。
- 不要为了“更漂亮”增加新交互。
- 如果发现某个页面需要新的布局语义，先回到这份任务单，不要临时起意。

## 验收清单

最终必须满足：

- `xcodebuild -project AcMind.xcodeproj -scheme AcMind -configuration Debug -destination 'platform=macOS' build` 通过
- 六个页面的壳层视觉一致
- 六个页面的主任务一眼可辨
- 六个页面没有彼此不兼容的容器语言
- 不出现多余滚动才能看到结论的情况
- 所有修改都能对应回这份任务单的条目
