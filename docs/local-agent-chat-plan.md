# AcMind 首页本地 Agent 对话界面从 0 到 1 实施方案

## 1. 目标

在 AcMind 的首页（图一）新增一个“对话界面”，让用户可以像图二、图三那样直接和本地 Agent 交流，并把对话结果用于“收集、整理、执行任务、回填结果”等场景。

这个界面不是单纯的聊天窗口，而是一个“任务驱动型对话面板”：

- 能连接你本地部署的 Agent
- 支持流式回复
- 支持任务状态展示
- 支持把首页里的内容上下文带入对话
- 支持把对话结果一键转成 AcMind 里的待办、暂存、整理任务

## 2. 推荐落点

建议不要把它做成一个完全独立的新产品页，而是优先放在首页内部，原因是：

- 首页本来就是“今天要处理什么”的总入口
- 首页已经有快速动作、暂存池、内容预览等模块，最适合承接 Agent 任务
- 右侧可扩展区域、底部卡片区都适合挂一个聊天面板

### 推荐布局方案

优先级从高到低：

1. 首页主内容区新增一个“对话面板”模块
2. 右侧 Inspector 增加一个“Agent”标签页
3. 额外保留一个独立 AI 聊天页，方便全屏任务处理

### 最推荐的组合

- 首页内嵌一个轻量版对话面板
- 独立的全屏 Agent 页面作为“展开模式”

这样既不破坏首页信息密度，也能保留完整对话体验。

## 3. 参考现状

AcMind 现在已经具备一些可复用基础：

- 首页页面：`src/renderer/pages/daily-flow/DailyKnowledgeFlowPage.tsx`
- 应用壳：`src/renderer/components/layout/AppShell.tsx`
- 右侧信息栏：`src/renderer/components/layout/RightInspector.tsx`
- AI Runtime：`src/renderer/pages/ai/AIPage.tsx`
- AI 调用层：`src/main/services/aiHub/aiProviderService.ts`
- AI 任务执行器：`src/main/services/aiHub/aiActionRunner.ts`
- 现有页面路由：`src/renderer/App.tsx`

因此本次新功能不建议从零重写，而是沿用现有架构，新增一个“Agent Chat”能力层。

## 4. 你最终会得到什么

用户在首页可以直接看到：

- 一个类似 ChatGPT 的输入框
- 消息流和回答流
- 当前任务状态：等待中、执行中、成功、失败
- 本地 Agent 正在使用的能力说明
- 一键插入当前选中内容作为上下文
- 一键把结果变成“继续处理 / 入库 / 生成待办”

在交互上，它应该像一个“工作台里的执行助手”，而不是普通聊天软件。

## 5. 核心交互设计

### 5.1 首页入口

首页顶部或主体区域增加一个入口，例如：

- `打开 Agent 对话`
- `开始小龙虾任务`
- `发起本地任务`

点击后打开：

- 首页内嵌面板
- 或右侧抽屉
- 或全屏聊天页

### 5.2 对话面板组成

建议拆成 5 个区块：

1. 会话标题区
2. 消息列表区
3. 输入与快捷指令区
4. 上下文挂载区
5. 任务状态区

### 5.3 支持的基础能力

- 发送文本消息
- 流式显示回复
- 中断生成
- 重试上一条
- 查看历史会话
- 附加首页当前内容作为上下文
- 将回复转成可执行任务
- 将结果保存为笔记、待办或暂存内容

## 6. 建议的功能范围

### MVP 必须做

- 新建会话
- 发送消息
- 连接本地 Agent
- 流式输出
- 消息历史记录
- 简单的任务状态展示
- 错误与重试
- 快捷指令模板

### 第二阶段再做

- 多会话管理
- 消息分支
- 任务计划树
- 文件上传
- 图片输入
- 工具调用详情
- Agent 运行日志
- 多 Agent 协同

## 7. 本地 Agent 接入方式

建议把接入方式设计成“可插拔适配器”，不要把接口写死。

### 7.1 最小兼容策略

优先支持这三类之一：

- OpenAI-compatible HTTP 接口
- SSE 流式接口
- WebSocket 实时接口

### 7.2 推荐的抽象层

在主进程新增一个本地 Agent 服务层，例如：

- `localAgentService`
- `agentChatService`
- `agentGateway`

它负责：

- 拼装请求
- 传递上下文
- 处理流式消息
- 中断任务
- 健康检查
- 统一错误格式

### 7.3 建议的配置项

在设置页新增本地 Agent 配置：

- Base URL
- 协议类型
- Token / API Key
- 默认 Agent 名称
- 超时时间
- 是否允许流式输出
- 默认任务模板

## 8. 数据结构建议

如果要做得完整，建议补三类数据：

### 8.1 会话表

记录一次完整对话：

- 会话 id
- 会话标题
- 创建时间
- 更新时间
- 关联页面
- 关联任务类型
- 当前状态

### 8.2 消息表

记录每条消息：

- 消息 id
- 会话 id
- 角色：user / assistant / tool / system
- 文本内容
- 流式状态
- 错误信息
- 时间戳

### 8.3 任务执行表

记录 Agent 的任务执行：

- 任务 id
- 对话 id
- 输入摘要
- 使用的 Agent
- 调用状态
- 开始与结束时间
- 错误信息
- 结果引用

## 9. 业务流程设计

### 9.1 基础流程

1. 用户在首页打开 Agent 对话面板
2. 用户输入“我要完成小龙虾任务”
3. 前端把消息发到主进程
4. 主进程转发给本地 Agent
5. Agent 流式返回结果
6. 前端实时显示
7. 结果生成可执行动作
8. 用户选择继续执行、保存或结束

### 9.2 带上下文流程

1. 用户在首页选中一条暂存内容或收集内容
2. 点击“发送给 Agent”
3. 当前内容作为上下文注入
4. Agent 根据上下文完成任务
5. 回写为整理建议、待办或入库内容

### 9.3 任务模板流程

可以先做一个固定模板：

- `小龙虾任务`

这个模板可以预设：

- 目标
- 约束
- 输出格式
- 结果落点

例如：

- 先理解需求
- 再列出执行步骤
- 再返回最终结果
- 最后生成可以保存到 AcMind 的结构化输出

## 10. 页面结构建议

### 10.1 首页内嵌版

建议在首页下半区新增一个模块：

- 左边：快速动作
- 中间：暂存池 / 今日内容
- 右边：Agent 对话面板

这个布局最适合图一当前结构。

### 10.2 独立展开版

当用户需要长对话时：

- 从首页点击“展开”
- 打开独立 AI Chat 页面

这个页面可以复用同一套消息组件，只是把容器做大。

### 10.3 右侧 Inspector 扩展版

如果你想尽量少改首页结构：

- 给右侧 Inspector 增加一个 `Agent` 标签
- 在里面放简化版对话与任务状态

这个方案改动最小，但输入体验会比首页内嵌稍弱。

## 11. 技术方案建议

### 11.1 前端

新增这些组件：

- `AgentChatPanel`
- `ChatMessageList`
- `ChatComposer`
- `ChatSessionSwitcher`
- `TaskStatusBadge`
- `ContextAttachmentBar`

建议放在：

- `src/renderer/components/agent/`
- 或 `src/renderer/pages/agent-chat/`

### 11.2 主进程

新增这些能力：

- `agentChat.createSession`
- `agentChat.sendMessage`
- `agentChat.cancelMessage`
- `agentChat.listSessions`
- `agentChat.getSession`
- `agentChat.healthCheck`

### 11.3 preload

通过 preload 暴露到 renderer：

- `window.acmind.agentChat.*`

### 11.4 存储

如果要持久化：

- 会话、消息、执行记录写入本地数据库
- 失败任务保留错误原因
- 生成结果支持再次打开

## 12. 从 0 到 1 的实施步骤

### 第 1 步：确认本地 Agent 协议

先确认你的本地 Agent 对外暴露什么接口：

- HTTP
- SSE
- WebSocket
- OpenAI-compatible

这是整个方案的第一关键点。

### 第 2 步：确定首页放置位置

建议优先定一个版式：

- 首页右下或底部新增 Agent 面板
- 或三栏中的最右一栏作为聊天区域

先定位置，后写交互，避免界面反复推翻。

### 第 3 步：先做 UI 骨架

先不接真实 Agent，只把界面做出来：

- 会话列表
- 消息流
- 输入框
- 发送按钮
- 中断按钮
- 空状态

这一步的目标是“看起来已经能用”。

### 第 4 步：接入本地 Agent 健康检查

先做一个最简单的连通性检查：

- Base URL 是否可达
- Agent 是否在线
- 是否需要鉴权

让用户一眼知道“现在能不能用”。

### 第 5 步：接入单轮消息

完成第一版发送与返回：

- 输入一条消息
- 得到一条回复

先不做复杂任务编排。

### 第 6 步：接入流式输出

把“回答一口气返回”改成“边生成边展示”：

- 增强对话体验
- 也方便用户提前中断

### 第 7 步：接入上下文

让 Agent 读取首页已有内容：

- 当前选中的暂存
- 当前待整理内容
- 当前页面摘要

这样它才真正变成“AcMind 的执行助手”。

### 第 8 步：加入任务模板

增加快捷入口：

- 小龙虾任务
- 总结当前内容
- 生成待办
- 生成整理建议

模板先做少量，避免一开始过度复杂。

### 第 9 步：加入持久化

把以下内容保存到本地：

- 会话
- 消息
- Agent 返回结果
- 错误日志

这样刷新页面后不会丢失对话。

### 第 10 步：做完成态与失败态

补齐以下体验：

- Agent 离线
- 连接超时
- 任务失败
- 重试失败
- 中断成功

### 第 11 步：联动 AcMind 现有能力

最终把对话结果接回 AcMind 的主流程：

- 生成暂存
- 推送到收集箱
- 触发整理
- 写入知识库
- 生成任务队列

## 13. 推荐里程碑

### Milestone A：可见

目标：

- 首页出现一个新的对话面板
- 能输入消息
- 能看到静态假数据回复

### Milestone B：可通

目标：

- 接通本地 Agent
- 能完成一问一答
- 能看到连接状态

### Milestone C：可用

目标：

- 流式输出
- 历史会话
- 任务模板
- 上下文挂载

### Milestone D：可融入工作流

目标：

- 结果能进入 AcMind 主流程
- 可以直接变成暂存、待办、整理建议

## 14. 验收标准

满足以下条件，就可以认为这个功能完成了从 0 到 1：

- 首页能看到对话界面
- 可以发送消息到本地 Agent
- Agent 回答能实时显示
- 能中断和重试
- 能带入首页上下文
- 能把结果落到 AcMind 任务流里
- 刷新后对话记录还在

## 15. 风险点

### 15.1 协议不统一

如果本地 Agent 的接口不稳定，建议先做适配层，不要让 UI 直接依赖具体协议。

### 15.2 首页拥挤

如果把聊天面板直接塞进首页，信息密度会变高，建议加“展开模式”。

### 15.3 上下文过大

不要一次把所有首页数据都塞给 Agent，应该做上下文裁剪和摘要。

### 15.4 结果不可执行

如果 Agent 只是返回文字，价值会打折。一定要规划“结果如何回到 AcMind 流程”。

## 16. 建议的实施顺序

如果按最稳妥的顺序做，我建议这样推进：

1. 先确认本地 Agent 接口
2. 再定首页对话面板的位置
3. 先做 UI 骨架
4. 再做单轮对话
5. 再做流式输出
6. 再做上下文接入
7. 再做任务模板
8. 再做持久化
9. 最后接回 AcMind 的收集、整理、入库流程

## 17. 建议的 MVP 范围

如果你想最快看到效果，MVP 只保留这 6 个点：

- 首页有一个 Agent 面板
- 能输入消息
- 能连上本地 Agent
- 能流式返回
- 能中断
- 能保存最近会话

这已经足够验证方向。
