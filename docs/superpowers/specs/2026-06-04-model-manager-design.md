# 模型管理页设计

## Goal

在 AcMind 主界面中新增一个原生页面「模型管理」，放在左侧导航的系统区，与「设置」同级可达。

这个页面的目标不是展示模型列表本身，而是把当前过多、过散的模型入口收拢到一个可快速筛选和切换的工作台，让用户能够在一个页面里完成以下动作：

- 快速看清当前有哪些模型
- 快速筛选出需要的模型
- 快速切换默认模型
- 快速判断某个模型属于语音识别还是语音克隆
- 快速判断某个模型是本地、云端，还是未配置/未下载

页面风格必须贴近 AcMind 现有的 macOS 原生界面，而不是网页式控制台。

## Current Problem

当前模型相关能力分散在多个位置：

- AI 提供者和模型默认值分散在设置与运行时路由里
- 语音识别模型分散在本地模型管理、STT 路由和云端转写器里
- 语音克隆没有独立的管理入口
- 用户想判断“现在到底该用哪个模型”时，需要在多个页面间来回切换

这会造成两个直接问题：

1. 模型太多，用户看不清。
2. 模型能力不同，但入口看起来像一堆同类项，容易混淆。

## Perfect State

这个页面在理想状态下应该长这样：

### 入口位置

- 左侧导航新增一个系统级条目：`模型管理`
- 位置与 `设置` 同级
- 点击后，右侧主内容区切换到模型管理页面
- 它是主界面的一部分，不是单独网页，也不是设置页的子弹窗

### 页面第一眼

- 顶部能看到当前模型管理页的摘要
- 一眼知道：
  - 默认语音识别模型是谁
  - 默认语音克隆模型是谁
  - 多少模型可用
  - 多少模型已下载
  - 多少模型是云端 API

### 页面操作

- 用户能通过筛选条件快速压缩范围
- 用户能搜索模型名称或 provider 名称
- 用户能在列表里直接切默认、启用、停用、下载、删除
- 用户能看出一个模型到底属于哪一类能力

### 分类清晰

必须把以下概念分开：

- 语音识别
- 语音克隆
- 本地模型
- 云端模型
- AI 通用模型

不能再让“识别模型”和“克隆模型”看起来像同一类东西。

### 风格

- 浅色、克制、留白充足
- 细边框、轻阴影、圆角卡片
- 字体和层级接近 macOS 系统设置页
- 不要做成网页后台风格

## Non-Goals

- 不在本次设计里重构所有模型系统
- 不把模型管理做成独立应用
- 不在这一步新增复杂的模型推荐算法
- 不要求本次完成所有交互实现，只要定义清楚页面和落点

## UX Structure

### 1. 左侧导航入口

在左侧主导航中新增 `模型管理`：

- 与 `设置` 同级
- 采用与现有导航一致的图标和行高
- 可以保留系统区风格，但要让它在视觉上像一个正式页面，而不是工具项

### 2. 顶部摘要区

建议放在页面顶部，内容包括：

- 当前默认语音识别
- 当前默认语音克隆
- 本地模型数量
- 云端模型数量
- 已启用模型数量

摘要区的作用是让用户不需要往下翻就知道当前状态。

### 3. 筛选区

建议提供这些筛选维度：

- 功能域
  - AI
  - 语音识别
  - 语音克隆
  - 其他
- 运行形态
  - 本地
  - 云端 API
  - 已下载
- 状态
  - 已启用
  - 可切换
  - 待下载
  - 未配置
- 搜索
  - 模型名
  - provider 名称
  - 关键词
- 排序
  - 当前启用优先
  - 本地优先
  - 低成本优先
  - 高质量优先

### 4. 模型列表区

列表推荐按“卡片 + 密集列表混合”的方式呈现：

- 被当前选中的模型可使用更明显的强调态
- 每个模型至少显示：
  - 名称
  - 功能域
  - 运行形态
  - 默认状态
  - 可用状态
  - 大小或成本提示
- 每个模型支持快捷动作：
  - 设为默认
  - 启用 / 停用
  - 下载 / 删除
  - 查看详情

### 5. 详情区

选中某个模型后，在右侧或下方显示更详细信息：

- 模型所属功能域
- 是否本地
- 是否云端
- 默认语言或适用语言
- 当前配置状态
- 相关 API Key 或下载状态
- 失败原因或不可用原因

## Data Model

这个页面不应该直接拼字符串，而应该有一个统一的视图模型层来聚合现有数据。

### Model Category

建议至少分这几类：

- `ai`
- `speechRecognition`
- `voiceClone`
- `localModel`
- `cloudModel`

### Model Item

每个模型条目建议包含：

- `id`
- `displayName`
- `category`
- `providerId`
- `modelId`
- `deploymentKind`
- `isDefault`
- `isEnabled`
- `isDownloaded`
- `requiresAPIKey`
- `sizeLabel`
- `statusLabel`
- `capabilities`

### Display Rules

- 语音识别和语音克隆必须独立成类目
- 本地与云端是“运行形态”，不是“能力类型”
- 一个模型可以同时拥有多个标签，但主分类必须唯一
- 如果一个模型当前不可用，要明确显示原因，而不是只灰掉

## Architecture

### 1. New Page Route

在主界面的页面路由中新增模型管理入口，让它成为一个正式的主页面。

### 2. Model Management ViewModel

新增一个专门的视图模型，用来聚合：

- AI provider 配置
- 语音识别 provider 配置
- 本地 ASR 模型状态
- 语音克隆 provider 状态
- 当前默认模型状态

这个视图模型负责：

- 拉取数据
- 计算筛选结果
- 维护当前选中条目
- 执行切换和管理动作

### 3. Reuse Existing Services

页面应尽量复用现有服务，而不是复制逻辑：

- `SettingsService`
- `AIRuntimeService`
- `STTRouter`
- `LocalASRManager`
- 现有 provider 存储结构

### 4. Read-only First, then Actions

设计上建议先让页面支持只读展示，再逐步接入动作：

- 第一步：展示统一列表和筛选
- 第二步：支持设为默认、切换状态
- 第三步：支持下载 / 删除 / 配置

这样可以降低页面在早期接入时的风险。

## State Flow

1. 主窗口加载时，左侧导航显示 `模型管理`。
2. 用户点击后，主内容区切换到模型管理页面。
3. 页面初始化时读取：
   - AI provider 配置
   - 语音识别 provider 配置
   - 本地模型下载状态
   - 当前默认模型
   - 语音克隆能力入口
4. 页面根据筛选器计算当前可见列表。
5. 用户选择某个模型后，详情区显示该模型的完整信息。
6. 用户执行切换动作后，页面刷新状态并反映真实运行结果。

## Error Handling

需要在页面中明确处理这些状态：

- 未配置 API Key
- 模型未下载
- 模型当前不可用
- 默认模型已失效
- 获取 provider 列表失败
- 获取本地模型状态失败

错误展示原则：

- 不要只给空白页
- 不要只灰掉按钮
- 要显示明确原因和可执行下一步

## Acceptance Criteria

这个页面达到“完美状态”时，至少满足以下条件：

- 左侧导航中存在 `模型管理`
- 它是主界面的原生页面，不是网页
- 页面内可以区分语音识别和语音克隆
- 页面内可以按本地、云端、已下载、状态进行筛选
- 页面内可以快速切换默认模型
- 页面能显示当前真实的模型状态
- 页面风格与 AcMind 的 macOS 原生界面一致

## Task Breakdown

### Phase 1: 页面定位

- 把 `模型管理` 加入主导航
- 定义它在路由里的位置
- 确保进入后显示在主内容区

### Phase 2: 数据聚合

- 汇总 AI provider、STT provider、本地 ASR 模型、语音克隆信息
- 统一成一个页面可消费的模型条目结构

### Phase 3: 页面布局

- 顶部摘要
- 左侧筛选
- 中间模型列表
- 右侧或下方详情区

### Phase 4: 动作接入

- 设默认
- 启用 / 停用
- 下载 / 删除
- 配置 API Key

### Phase 5: 风格统一与验证

- 调整成 Apple 原生风格
- 检查空状态、错误状态和加载状态
- 验证不会影响现有设置和语音入口

## Verification

完成设计后，下一步实现时至少要验证：

- 页面能从主导航正常进入
- 模型筛选可用
- 默认模型切换后真实生效
- 语音识别和语音克隆入口没有混淆
- 页面风格与现有原生 UI 保持一致
